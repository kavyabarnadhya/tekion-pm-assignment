Payment Link Generation Workflow

An orchestration layer that lets dealerships generate a shareable payment link for a customer directly from their management system. It receives the request, calls a Payment Link API, and writes the result back to the payment record.

1Business problem

Dealerships currently lack the ability to generate shareable payment links for customers directly within their management system. This creates friction in the payment collection process: a customer service order gets closed, but collecting payment still means a manual, out-of-system step.

Tekion aims to bridge this gap by integrating with a payment link generation service through an orchestration layer, so a payment link can be generated and the record updated without leaving the dealership's system of record.

2High-level PRD

Objective: Configure an integration workflow that receives payment request data from Tekion, generates a payment link via an external API, and updates the payment record in the system.

Scope:

  • Receive payment request data from Tekion (customer info, amount, invoice number)
  • Call the Payment Link API to generate a payment link
  • Extract response fields (payment_link, ID, referenceId)
  • Update the payment record via the Update API
  • Implement error handling for all failure scenarios

Success criteria: all five scope items working end-to-end against the provided mock APIs, with idempotency (no duplicate links on retried requests) and graceful, logged failure handling rather than silent drops.

3Architecture

flowchart TD A[Webhook: Tekion Payment Request] --> B{Required fields present?} B -- No --> B1[Log failed record\nstatus: failed] B -- Yes --> C[Data Store: Get Record by referenceId] C --> D{payment_link already exists?} D -- Yes --> D1[Idempotent End\nreturn existing link] D -- No --> E[Map fields: flat -> nested\ncompute expire_by] E --> F[HTTP: Call Payment Link API] F -- permanent failure / retries exhausted --> F1[Log failed record] F -- success --> G[Parse response:\npayment_link, ID, referenceId] G --> H[HTTP: Call Update API] H -- permanent failure / retries exhausted --> H1[Log failed record] H -- success --> I[Data Store: Upsert record\nstatus: updated]

4Key design decisions

Idempotency by referenceId

referenceId is the idempotency key for the whole workflow. Before calling the Payment Link API, the scenario looks up any existing record for that reference. If one already has a payment_link, it returns the existing link instead, so a retried or duplicate Tekion request can't generate two live payment links for the same invoice.

GET-with-body on the Update API

The spec defines the Update API as a GET request carrying a JSON body, which is non-standard REST. The workflow mirrors this exactly instead of quietly substituting a POST, and calls out the tension rather than pretending it's not there.

Retry via Make's native mechanism

Instead of a custom retry-counter loop, the scenario uses Make's Store incomplete executions setting: automatic exponential backoff on transient failures (timeouts, 5xx), straight through to error-handler logging on permanent failures (4xx). That's less code for a mock API that isn't flaky in a way a hard attempt cap would actually improve.

Flat-to-nested field mapping

Tekion's flat payload (customerName, customerPhone, customerEmail) gets mapped into the Payment Link API's nested customer object. Optional fields like email are left out entirely rather than sent as null when absent.

5Tested results

✓
Happy path: full valid payload ran end-to-end, PaymentRecords updated with a real payment_link and status: updated.
✓
Missing-field validation: payload with a required field dropped correctly routed to failure logging, downstream modules skipped.
✓
Idempotency replay: re-sending the same referenceId stopped at the lookup step, Payment Link API wasn't called a second time.
✓
API-failure error handling: a real Payment Link API failure was caught and logged with status: failed and an error reason.
